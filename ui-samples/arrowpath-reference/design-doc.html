◆ Executive Summary

ArrowFleet Services is a transport and logistics company based in Kitengela and Limuru, Kenya, operating ~50 mixed vehicles (trucks, pickups, SUVs, vans, heavy machinery transporters). The company offers all kinds of vehicle transport — serving industries (EPC & construction), ferrying visitors and travellers from airports, and moving materials and cargo between locations across Kenya.

This platform is a Progressive Web App (PWA) — installable on mobile devices and desktops for offline-capable, native-like access. It provides centralized control over driver operations, vehicle fleet management, maintenance & spare parts tracking, financial operations, customer/client service, and real-time vehicle tracking — replacing fragmented manual processes with a unified digital dashboard.

The platform serves fleet administrators who oversee the entire operation: managing driver onboarding and verification, dispatching vehicles to project sites and airport pickups, monitoring fleet movements in real-time via GPS tracking, scheduling preventive maintenance and spare parts procurement, processing financial transactions (revenue, payouts, client invoicing), handling service requests, and analyzing operational performance through analytics dashboards.

What Problem Does This Solve?

Fleet operators currently juggle multiple disconnected tools — spreadsheets for driver records, scattered WhatsApp groups for dispatch coordination, paper logbooks for maintenance, separate accounting for payouts, and phone-based client support. ArrowFleet eliminates WhatsApp dependency entirely with built-in broadcast messaging, direct chat, and an activity feed — consolidating everything into a single installable PWA where an administrator can onboard a new driver, assign them a vehicle, dispatch to a project site or airport pickup, track vehicle location, schedule maintenance, manage spare parts inventory, process earnings, and handle client requests — all without leaving the dashboard.

What Makes This Special

ArrowFleet is purpose-built for commercial fleet operations — not consumer ride-hailing. The operator-centric model means features like vehicle maintenance scheduling, spare parts inventory, driver document/license compliance, fleet-level financial reporting with client invoicing, and KRA tax reporting are first-class concerns. The fleet handles diverse vehicle types (from saloon cars to heavy-duty trucks), scheduled project-site dispatches, airport transfers, cargo/materials transport, multi-day assignments, and route-based operations between Kitengela, Limuru, Nairobi airports (JKIA/Wilson), and client locations across Kenya. As a PWA, the app works offline, can be installed on any device, and delivers push notifications for dispatch alerts and fuel warnings.

➔ User Flows & Journey Maps

Flow A: Driver Onboarding

  1. Admin creates driver profile (name, phone, email)
  2. Driver uploads documents (license, ID, insurance)
  3. Admin reviews & verifies each document
  4. Background check initiated (manual or automated)
  5. Admin assigns vehicle from fleet inventory
  6. Driver completes training modules (if required)
  7. Driver status set to Active — can accept rides

Flow B: Ride Lifecycle

  1. User requests ride (via app or admin enters booking)
  2. System creates Booking (status: Pending)
  3. Admin/system assigns driver & vehicle → Booking: Confirmed
  4. Driver heads to pickup → Ride: Picking Up
  5. Passenger picked up → Ride: In Progress
  6. Arrive at destination → Ride: Completed
  7. Payment processed → Transaction created
  8. Passenger rates driver & leaves feedback

Flow C: Driver Payout

  1. Rides accumulate in driver’s earnings ledger
  2. Platform commission deducted per ride (configurable %)
  3. Admin triggers payout (manual or scheduled batch)
  4. M-Pesa B2C disbursement initiated
  5. Transaction status: Pending → Completed / Failed
  6. Driver receives SMS confirmation

Flow D: Support Ticket Resolution

  1. Passenger submits complaint (linked to ride)
  2. Ticket created: Open, priority auto-assigned
  3. CS agent picks up ticket → In Progress
  4. Agent contacts driver/passenger, investigates
  5. Resolution applied (refund, warning, etc.)
  6. Ticket: Resolved → auto-closes after 48h

Flow E: Document Expiry Renewal

  1. System detects document expiring in 30 days
  2. Notification sent to admin + driver
  3. Reminder at 14 days and 7 days
  4. Driver uploads renewed document
  5. Admin reviews & verifies → Verified
  6. If expired without renewal → driver auto-suspended

Flow F: Admin Login & Session

  1. Admin navigates to /auth/login
  2. Enters email + password
  3. 2FA challenge (OTP via SMS/authenticator)
  4. Session created (JWT or server session)
  5. Redirected to Dashboard
  6. Auto-timeout after 30 min inactivity

🔒 Authentication & Permissions Matrix

Authentication Spec

  • Login: Email + password with bcrypt hashing (min 12 chars, 1 uppercase, 1 number, 1 symbol)
  • 2FA: TOTP (authenticator app) or SMS OTP for all admin roles
  • Sessions: Server-side sessions or signed JWTs; 30-minute inactivity timeout; max 3 concurrent sessions
  • Password Reset: Email-based reset link (15-minute expiry, single-use token)
  • Account Lockout: 5 failed attempts → 15-minute lockout → admin unlock required after 3 lockouts
  • Session Revocation: Sign out invalidates token immediately; admin can force-logout any user

⚙ State Machines & Lifecycle Transitions

Valid state transitions for each core entity. Arrows show allowed transitions; any transition not listed is invalid.

Ride Lifecycle

Scheduled → Picking Up → In Progress → Completed ↓ ↓ ↓ Cancelled Cancelled Cancelled (with partial fare) Rules: • Scheduled → Cancelled: free cancellation (no charge) • Picking Up → Cancelled: cancellation fee may apply • In Progress → Cancelled: partial fare charged based on distance • Completed is terminal — no further transitions • Cancelled is terminal — no further transitions

Booking Lifecycle

Pending → Confirmed → Scheduled → Completed ↓ ↓ ↓ Cancelled Cancelled Cancelled Rules: • Pending → Confirmed: when driver & vehicle assigned • Confirmed → Scheduled: when ride time is set/approaching • Scheduled → Completed: when linked Ride completes • Any non-terminal state → Cancelled: allowed with reason

Driver Status Lifecycle

Pending Verification → Active ↔ Inactive ↓ ↓ ↓ Rejected Suspended Suspended ↓ Active (after review) Rules: • Pending → Active: all documents verified, training complete • Pending → Rejected: failed verification (terminal unless re-applies) • Active ↔ Inactive: driver toggles availability or admin disables • Active/Inactive → Suspended: policy violation, expired docs, or complaints • Suspended → Active: admin review & reinstatement only

Vehicle Status Lifecycle

Available ↔ In Use ↓ ↓ Maintenance Maintenance (emergency) ↓ Available (after service) or Retired (permanent) Rules: • Available → In Use: assigned to active ride • In Use → Available: ride completed • Any → Maintenance: scheduled service or breakdown • Maintenance → Available: service completed • Any → Retired: vehicle decommissioned (terminal)

Document Verification Lifecycle

Pending → Verified → Expired (auto, on expiry date) ↓ ↓ Rejected Pending (new upload) Rules: • Pending → Verified: admin approves • Pending → Rejected: admin rejects (with reason) • Rejected → Pending: driver re-uploads corrected document • Verified → Expired: automatic when expiry_date passes • Expired → Pending: driver uploads renewal

Payment Transaction Lifecycle

Pending → Completed ↓ ↓ Failed Refunded Rules: • Pending → Completed: payment gateway confirms success • Pending → Failed: gateway timeout, insufficient funds, etc. • Completed → Refunded: admin initiates refund (partial or full) • Failed → Pending: retry allowed (max 3 attempts)

Key Relationships

  • A Driver is assigned to one Vehicle (1:1 at any time)
  • A Ride/Trip connects one Driver, one Client, and one Vehicle
  • A Booking maps to one future Ride/Trip
  • A Driver has many Documents
  • A Ride generates one Transaction (payment) and optionally one (payout)
  • A Support Ticket references one Client and optionally one Ride
  • A Vehicle has many Work Orders and Maintenance Schedules
  • A Work Order consumes many Spare Parts (deducted from inventory)
  • A Spare Part is supplied by one or more Suppliers
  • A Vehicle has many Fuel Logs and Inspections
  • A Vehicle has one GSU Device that produces many GSU Fuel Readings (time-series telemetry)

⚙ API Specification

API Design Principles

  • Style: RESTful JSON API with consistent resource naming
  • Versioning: URL-based (/api/v1/); breaking changes require new version
  • Authentication: Bearer token (JWT) in Authorization header
  • Rate Limiting: 100 requests/minute per user; 1,000/minute for service accounts
  • Pagination: Cursor-based for large collections; default page size 25, max 100
  • Error Format: Consistent JSON: { "error": { "code": "DRIVER_NOT_FOUND", "message": "...", "status": 404 } }

🚀 Migration Plan (V1 → V2)

Current State (V1)

  • Static HTML/CSS/JS prototype deployed on Netlify (arrowfleet-v1.netlify.app)
  • No backend, no database, no authentication — purely presentational
  • Demonstrates UI structure and navigation for 23+ pages
  • No real data; sample/mock data hardcoded in HTML

Migration Strategy

  1. Design System Extraction: Extract color palette, typography, spacing, and component patterns from V1 into a Tailwind/shadcn design token file
  2. Convex Backend Setup: Define Convex schema, tables, and indexes; create seed scripts with realistic sample data matching V1 patterns
  3. Auth System: Implement login, RBAC, and session management (no user data to migrate — start fresh)
  4. Core Modules Build: Rebuild each module in Next.js/React connected to Convex backend, following V1 UI as reference
  5. Data Entry: Manual entry of existing fleet data (drivers, vehicles) via admin UI or CSV import tool
  6. Integration: Connect M-Pesa Daraja, Kopokopo, Maps, HostPinnacle SMS APIs
  7. UAT: Parallel run — V1 as reference, V2 for real operations; validate all screens match or improve V1
  8. Go-Live: DNS switch; decommission V1 Netlify site; begin live operations
