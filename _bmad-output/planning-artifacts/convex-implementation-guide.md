# Convex Implementation Guide — SommyComfort

This guide consolidates the verified Convex research for SommyComfort's backend, which is built on `convex@1.40.0` (installed at `packages/backend/convex`). It supersedes the original NestJS/Redis/BullMQ/S3/Socket.IO stack per the architecture **Backend Platform Addendum** (AR3′/AR4′/AR6′/AR7′): Convex queries/mutations/actions replace REST controllers, the built-in reactive query system replaces the Socket.IO gateway, `ctx.scheduler` + crons replace Redis + BullMQ, `ctx.storage` replaces S3/MinIO, and Convex Auth replaces JWT + argon2id. All API facts below are confirmed against the installed package's type declarations unless explicitly flagged; `@convex-dev/auth` and `convex-test` are **not yet installed** and their snippets are confirmed against current docs only. Each section ends with concrete gotchas to verify against the live dev deployment.

## Conventions

- **Function-file layout under `convex/`**: one file per domain area (e.g. `auth.ts`, `payments.ts`, `mpesa.ts`, `guestDocuments.ts`, `housekeeping.ts`); shared helpers in `convex/lib/` (e.g. `lib/auth.ts`); the inbound webhook router in `convex/http.ts`; recurring jobs in a single `convex/crons.ts`; the schema in `convex/schema.ts`. Tests live beside the functions they cover (e.g. `auditLogs.test.ts`).
- **Naming**: public functions are `query`/`mutation`/`action`; internal-only functions are `internalQuery`/`internalMutation`/`internalAction` (prefer internal for anything not called directly by a client — scheduled targets, fan-out workers, webhook handlers). Tables and fields are `camelCase`; relations are `<entity>Id` fields typed `v.id("<table>")` plus an `.index("by_<field>", [...])` — there are no foreign keys (AR4′).
- **`v.*` validators are the server contract**: every function declares `args` (and, where useful, `returns`) with `convex/values` validators, and `schema.ts` validates every table. The validators are the authoritative input/output contract — the web client mirrors them for UX only and is never trusted.
- **Money is `v.int64()` integer minor units (cents)** — never floats (AR4′, NFR14). M-Pesa `Amount` is whole KES; convert to/from minor units and reject non-whole-shilling amounts at the boundary.
- **Audit is written from mutations**: every state-changing mutation writes an `auditLogs` row (AR9) within the same transaction, so the audit trail is atomic with the change. `auditLogs.actorId` derives from `ctx.auth` (becomes `v.id("users")` once `users` exists), not from client args.
- **Error handling**: throw on the failure path (`UNAUTHENTICATED`, `FORBIDDEN: <area>:<action>`, validation failures). Mutations are single ACID transactions that roll back on throw; make all scheduled/replayable handlers idempotent by guarding on current state, since Convex has no DB-level unique constraints. Webhooks always respond `200` fast and do the work in a mutation.

## Auth & RBAC

### 1. Identity — Convex Auth password provider

Source: installed `convex/server` (`Auth.getUserIdentity(): Promise<UserIdentity|null>`) + docs.convex.dev / labs.convex.dev/auth (current).

```ts
// convex/auth.ts
import { convexAuth } from "@convex-dev/auth/server";
import { Password } from "@convex-dev/auth/providers/Password";
export const { auth, signIn, signOut, store, isAuthenticated } = convexAuth({
  providers: [Password], // email+password; Convex handles hashing/sessions
});
```

- `convex/auth.config.ts` is also generated by the auth CLI (`npx @convex-dev/auth`) and registers the provider as a JWT issuer for `ctx.auth`.
- In a function, get the signed-in user with **`getAuthUserId(ctx)`** (`@convex-dev/auth/server`) → `Promise<Id<"users"> | null>`. The lower-level `ctx.auth.getUserIdentity()` (confirmed in the installed `authentication.d.ts`) returns OIDC claims (`subject`, `email`, …) and is what `getAuthUserId` wraps — prefer `getAuthUserId` so you get a typed `Id<"users">`.

### 2. Tables — RBAC from data-model.md (R1 Identity & Access)

`authTables` (spread from `@convex-dev/auth/server`) provides the `users`/`authSessions`/`authAccounts` tables. Re-express the data-model.md join entities as Convex tables (AR4′: relations = `v.id(...)` + index, not FKs). Drop `RefreshToken` (sessions are managed by Convex Auth).

```ts
// convex/schema.ts (added per-story when auth lands)
import { authTables } from "@convex-dev/auth/server";
export default defineSchema({
  ...authTables,                       // users, authSessions, authAccounts...
  // extend the auth `users` row with app fields (fullName, phone, isActive)
  roles:           defineTable({ name: v.string(), description: v.optional(v.string()) })
                     .index("by_name", ["name"]),
  permissions:     defineTable({ area: v.string(), action: v.union(v.literal("read"), v.literal("write"), v.literal("manage")) })
                     .index("by_area_action", ["area", "action"]),       // ↔ Permission @@unique([area,action])
  rolePermissions: defineTable({ roleId: v.id("roles"), permissionId: v.id("permissions") })
                     .index("by_role", ["roleId"]),                       // ↔ RolePermission
  userRoles:       defineTable({ userId: v.id("users"), roleId: v.id("roles") })
                     .index("by_user", ["userId"]).index("by_role", ["roleId"]), // ↔ UserRole
});
```

Note `auditLogs.actorId` (already in schema as `v.optional(v.string())`) should become `v.optional(v.id("users"))` once `users` exists.

### 3. `requirePermission` helper (in-function enforcement, AR6′)

```ts
// convex/lib/auth.ts
import { getAuthUserId } from "@convex-dev/auth/server";
import type { QueryCtx, MutationCtx } from "../_generated/server";

export async function requirePermission(
  ctx: QueryCtx | MutationCtx, area: string, action: "read" | "write" | "manage",
) {
  const userId = await getAuthUserId(ctx);
  if (!userId) throw new Error("UNAUTHENTICATED");
  const userRoles = await ctx.db.query("userRoles")
    .withIndex("by_user", q => q.eq("userId", userId)).collect();
  for (const ur of userRoles) {
    const perms = await ctx.db.query("rolePermissions")
      .withIndex("by_role", q => q.eq("roleId", ur.roleId)).collect();
    for (const rp of perms) {
      const p = await ctx.db.get(rp.permissionId);
      if (p && p.area === area && p.action === action) return userId; // granted
    }
  }
  throw new Error(`FORBIDDEN: ${area}:${action}`);
}
```

Call it as the first line of every protected `query`/`mutation`/`action` (e.g. `await requirePermission(ctx, "bookings", "write")`). The web mirrors permissions for UI gating only — the server stays authoritative.

### Gotchas / what to verify

- **Confirm the installed-version API** after `pnpm add`: read `node_modules/.pnpm/@convex-dev+auth@*/.../server` to verify `convexAuth`, `getAuthUserId`, and `authTables` signatures (they evolved across 0.0.x releases). The snippets above match current docs but are **not** yet verifiable against an installed package.
- `@convex-dev/auth` requires a peer `@auth/core` (pin the version its README specifies) and a `SITE_URL` + auth env keys (`JWKS`, `JWT_PRIVATE_KEY`) generated by `npx @convex-dev/auth`.
- The helper does N+1 `ctx.db.get`s per permission; acceptable at MVP role counts, but consider caching/denormalizing `area:action` strings onto `userRoles` if hot.
- Actions don't have `ctx.db` — for `action`-context permission checks, call a helper `query`/`mutation` via `ctx.runQuery`, or enforce in the calling mutation.
- `manage` should be treated as implying `read`/`write` for its area *only if* the PRD says so — current data-model treats them as discrete rows; confirm the semantics before adding implication logic.
- `getUserIdentity()` returns `null` (not throw) in queries/mutations but **throws** in HTTP actions — branch accordingly in `convex/http.ts` (e.g. M-Pesa webhook).

## File Storage

Verified against the installed package `convex@1.40.0` (`server/storage.d.ts`). Replaces S3/MinIO signed URLs per the Backend Platform Addendum (AR7′). No file-storage-specific changes in the 1.40.0 changelog — the API below is stable.

### Recommended upload flow (client → Convex)

Three steps: a mutation mints a one-time upload URL, the client POSTs the file directly to Convex storage, then a second mutation persists the returned `storageId`.

```ts
// packages/backend/convex/guestDocuments.ts
import { mutation, query } from "./_generated/server";
import { v } from "convex/values";

export const generateUploadUrl = mutation({
  args: {},
  returns: v.string(),
  handler: async (ctx) => {
    // TODO: assert caller is authed staff (RBAC) before issuing the URL
    return await ctx.storage.generateUploadUrl(); // short-lived POST URL
  },
});

export const saveGuestDocument = mutation({
  args: {
    guestId: v.id("guests"),
    kind: v.union(v.literal("id_front"), v.literal("id_back")),
    storageId: v.id("_storage"),
  },
  handler: async (ctx, { guestId, kind, storageId }) => {
    const meta = await ctx.db.system.get("_storage", storageId); // { size, contentType, sha256 }
    if (!meta) throw new Error("upload not found");
    if (meta.size > 10 * 1024 * 1024) throw new Error("file too large");      // size guard
    if (!["image/jpeg", "image/png", "application/pdf"].includes(meta.contentType ?? ""))
      throw new Error("unsupported type");                                     // content-type guard
    return await ctx.db.insert("guestDocuments", { guestId, kind, storageId });
  },
});
```

Client: `const url = await generateUploadUrl(); const { storageId } = await (await fetch(url, { method: "POST", headers: { "Content-Type": file.type }, body: file })).json(); await saveGuestDocument({ guestId, kind, storageId });`

### Reads + access control

`ctx.storage.getUrl(storageId)` returns a temporary HTTPS GET URL (or `null` if deleted). **Access control is your responsibility**: gate it inside an authed query, never expose `generateUploadUrl`/`getUrl` to anonymous callers, and never store the URL — generate it per request.

```ts
export const getGuestDocumentUrl = query({
  args: { docId: v.id("guestDocuments") },
  handler: async (ctx, { docId }) => {
    // require authed staff with the right permission here (RBAC, AR9)
    const doc = await ctx.db.get(docId);
    return doc ? await ctx.storage.getUrl(doc.storageId) : null;
  },
});
```

### Schema mapping (data-model.md → Convex)

Tables are added per-story (`schema.ts` convention). Replace the Prisma `objectKey String // S3 key` columns with `storageId: v.id("_storage")`:

- **Epic 4 — `GuestDocument`** (data-model.md line 238, `kind = id_front|id_back`, was `objectKey`): `guestDocuments` table with `guestId: v.id("guests")`, `kind`, `storageId: v.id("_storage")`, `.index("by_guest", ["guestId"])`. ID documents are sensitive — keep `getUrl` behind strict RBAC and audit access (AR9 `auditLogs`).
- **Epic 7 — `TaskPhoto`** (data-model.md line 345, R2 housekeeping, was `objectKey`): `taskPhotos` table with `taskId: v.id("housekeepingTasks")`, `storageId: v.id("_storage")`, `.index("by_task", ["taskId"])`. Mobile-camera capture → same upload flow.

### Gotchas / what to verify

- **Validate after upload, not before.** The client POSTs straight to Convex; the mutation can only inspect via `ctx.db.system.get("_storage", id)` afterward. Reject + `ctx.storage.delete(id)` on a bad file. Client `Content-Type` header on the POST is what populates `meta.contentType` — it's client-supplied, so treat as a hint and (for ID docs) consider server-side re-validation in an action.
- **`getMetadata()` and string `storageId` overloads are deprecated** in 1.40 — use `ctx.db.system.get("_storage", id)` and pass `v.id("_storage")`, not `v.string()`.
- **`store()`/`get()` (Blob) exist only in actions/HTTP actions**, not mutations — use them only if server generates files (e.g. PDF receipts), otherwise prefer the upload-URL flow.
- **Orphans/lifecycle:** deleting a `guestDocuments`/`taskPhotos` row does NOT delete the blob — call `ctx.storage.delete(storageId)` in the same mutation. Files uploaded but never saved are orphaned; add a cleanup cron if abuse is a concern.
- **Verify:** RBAC guard wired into `generateUploadUrl`/`getUrl`/`saveGuestDocument` before shipping; `_storage` reads use `ctx.db.system` (not `ctx.db`); `convex-test` covers the validation branches.

## M-Pesa (Convex actions + HTTP webhook)

**Source basis:** API confirmed against the installed `convex@1.40.0` — `httpRouter`/`httpAction` (`server/router.d.ts`), `ctx.scheduler.runAfter/runAt` and its retry semantics (`server/scheduler.d.ts`), and `action`/`internalAction` registration. Mapped onto `_bmad-output/planning-artifacts/mpesa-daraja-integration-spec.md`, `data-model.md` (`Payment`/`LedgerEntry`), and the architecture.md Backend Platform Addendum (AR3′/AR6′/AR7′). The Daraja flow itself (endpoints, OAuth, password) is from the spec — verify paths against `developer.safaricom.co.ke` at integration time.

**Version caveat (load-bearing):** Convex's per-function execution model **replaces BullMQ/Redis**. Confirmed in `scheduler.d.ts`:

- Scheduled **mutations** run **exactly once** (auto-retried on transient errors).
- Scheduled **actions** run **at-most-once** (NOT retried).

So put all DB writes in mutations and treat actions as best-effort. `fetch` is allowed in `action`/`httpAction` only (never in queries/mutations — they're deterministic & sandboxed).

### Recommended pattern

**1. STK initiation — `internalAction` (external fetch + OAuth).** Spec §2–§3. The client calls a public `mutation` that validates/normalizes the phone and amount (whole KES — reject non-integer shillings), inserts the `pending` payment, then `scheduler.runAfter(0, internal.mpesa.stkPush, {...})` to do the external call. Keep secrets (`MPESA_*`) in Convex env vars, never client-side.

```ts
// convex/mpesa.ts
export const stkPush = internalAction({
  args: { paymentId: v.id("payments"), phone: v.string(), amount: v.number(), reference: v.string() },
  handler: async (ctx, a) => {
    const token = await getOAuthToken();                  // Basic base64(key:secret) -> Bearer
    const res = await fetch(`${BASE}/mpesa/stkpush/v1/processrequest`, { /* body per spec §3 */ });
    const body = await res.json();
    await ctx.runMutation(internal.payments.recordStkAccept, {
      paymentId: a.paymentId,
      checkoutRequestId: body.CheckoutRequestID,
      merchantRequestId: body.MerchantRequestID,
      accepted: body.ResponseCode === "0",               // accepted != paid
    });
  },
});
```

OAuth token: no Redis — cache in a small `mpesaTokens` table (or fetch per-action; the addendum drops Redis). `ResponseCode === "0"` only means the push was accepted; post nothing to the ledger yet.

**2. Idempotent callback — `httpAction` in `convex/http.ts`.** Spec §4, AR3′ ("HTTP actions are used only for inbound webhooks"). The default export must be the `httpRouter()`. Respond `200 {ResultCode:0}` fast; do the ledger work in a mutation.

```ts
// convex/http.ts
import { httpRouter } from "convex/server";
import { httpAction } from "./_generated/server";
const http = httpRouter();
http.route({
  path: "/mpesa/callback",                      // embed/verify MPESA_CALLBACK_TOKEN (path or header) — spec §7
  method: "POST",
  handler: httpAction(async (ctx, req) => {
    const cb = (await req.json()).Body.stkCallback;        // Zod-validate shape first
    await ctx.runMutation(internal.payments.applyMpesaCallback, {
      checkoutRequestId: cb.CheckoutRequestID, resultCode: cb.ResultCode,
      resultDesc: cb.ResultDesc, metadata: cb.CallbackMetadata?.Item ?? [],
    });
    return new Response(JSON.stringify({ ResultCode: 0, ResultDesc: "Accepted" }), { status: 200 });
  }),
});
export default http;                            // must be default export
```

**3. Transactional persistence — `internalMutation`.** This is the idempotency + ledger boundary. The whole mutation is one ACID transaction (AR4′), which **replaces the BullMQ idempotent job** in spec §4:

- Look up `payment` by index on `provider_checkout_request_id`. If absent → log + flag for reconciliation (do not 4xx Daraja).
- If `status` is already `confirmed`/`failed` → **no-op return** (idempotent on Daraja retries — the natural idempotency key is `CheckoutRequestID`).
- On `ResultCode === 0`: set `confirmed`, store `MpesaReceiptNumber`/paid amount/phone/`paidAt`, insert the `LedgerEntry` payment row (sign convention per data-model), and write an `auditLogs` row (AR9) — all atomically. If paid `Amount` ≠ requested → still confirm but set `reconciled: false` for Story 5.8.
- On `ResultCode != 0` → `failed` + `resultDesc`, no ledger change.

**4. Status-query fallback + reconciliation.** Spec §5: instead of a BullMQ timeout job, in the accept mutation call `scheduler.runAfter(90_000, internal.mpesa.queryStkStatus, {paymentId})`. That action calls `stkpushquery/v1/query`; cap retries (~3 over 5 min — re-schedule from within, since actions don't auto-retry) then leave `pending` + flag. A `convex/crons.ts` daily job can sweep stale `pending` payments and unreconciled manual entries (spec §6).

### Maps to SommyComfort

- `Payment` (data-model.md §"R1 — Payments"): `provider` (`mpesa_stk`/`mpesa_manual`/`cash`/`card`), `status`, `amountCents` (`v.int64()` — never floats), `provider_checkout_request_id`, `provider_merchant_request_id`, `provider_receipt_number`, `paid_phone`, `paid_at`, `reconciled`. Add `.index("by_checkout", ["provider_checkout_request_id"])` for callback matching.
- The `@@unique([provider, providerReceiptNumber])` dedupe (manual M-Pesa, spec §6) has **no native unique constraint in Convex** — enforce in-mutation by querying an index before insert (see gotchas).
- `LedgerEntry` posting + computed booking balance = the shared money util (NFR14). `auditLogs` table already exists (AR9).

### Gotchas / what to verify

- **No DB unique constraints in Convex** — uniqueness (confirmed receipt numbers, single confirm per checkoutId) must be enforced inside the mutation via an index read in the same transaction. Verify the index exists.
- **Actions are at-most-once** — never put writes directly in `stkPush`; always hand off to a mutation via `ctx.runMutation`. Retries must be re-scheduled explicitly.
- **httpAction `ctx` has no `ctx.db`** — it can only `runQuery`/`runMutation`/`runAction`. Keep the webhook handler thin.
- **Webhook auth**: Convex HTTP action URLs are `https://<deployment>.convex.site/...` — set `MPESA_CALLBACK_URL` to that `.convex.site` host and verify `MPESA_CALLBACK_TOKEN` inside the handler (Convex can't IP-allowlist; do shared-secret + Zod shape check).
- **`int64` ↔ Daraja**: `amountCents` is `bigint`; Daraja `Amount` is whole KES. Convert and reject non-whole-shilling amounts before STK (spec §3).
- Verify Daraja endpoint paths/field names against the live portal (spec banner) and confirm PayBill-vs-Till `TransactionType` (spec §10 open item — Brian's decision).
- For tests: `convex-test` can drive mutations/queries; stub Daraja `fetch` in the action; one flagged sandbox smoke test (spec §9).

## Realtime & Offline

**Convex version: 1.40.0** (`node_modules/.pnpm/convex@1.40.0_react@19.2.4`; `convex/react` requires React 19, which matches `apps/web`). API below verified against the installed `dist/*-types` declarations.

### Reactive queries replace the Socket.IO gateway (AR3′)

Every Convex `query` is a live subscription: `useQuery` opens a WebSocket subscription, returns `undefined` while loading, and **re-renders automatically** whenever any document the query read changes. This is the built-in replacement for the superseded Socket.IO rooms-by-property+role gateway. There is no manual event emit/subscribe and no reconnect-reconciliation code to write.

```tsx
// apps/web — wrap once (NEXT_PUBLIC_CONVEX_URL), then in components:
import { useQuery } from "convex/react";
import { api } from "@/../packages/backend/convex/_generated/api";
const tasks = useQuery(api.housekeeping.listByAssignee, { status: "pending" }); // re-renders on change
if (tasks === undefined) return <Skeleton/>;          // loading
// pass "skip" to disable: useQuery(api.x.get, id ? { id } : "skip")
```

**Maps to SommyComfort:** the live surfaces from PRD §6 (housekeeping task queue, kitchen `RestaurantOrder` lanes, front-desk calendar, dashboards; data-model.md `HousekeepingTask`, `RestaurantOrder`/`OrderItem`, `Booking`) become plain `useQuery` calls — satisfies the "dashboard reflects change < 5 s" target without polling. TanStack Query is **no longer** the server-state layer for Convex data; keep it only for non-Convex fetches.

### Client cache + reconnection behavior

`ConvexReactClient` keeps a local query-result cache and one shared WebSocket. On disconnect it **auto-reconnects with backoff and resubscribes** all active queries, then transitions the UI to fresh results. Observe (don't reimplement) connection status via `useConvexConnectionState()` → `ConnectionState`: `{ isWebSocketConnected, hasInflightRequests, inflightMutations, inflightActions, connectionRetries, hasEverConnected, timeOfOldestInflightRequest }`.

### Optimistic updates

`useMutation(...).withOptimisticUpdate(...)` applies a synchronous local edit to cached query results, then auto-rolls-back when the server transition lands. Handler must be **synchronous** (returning a Promise is a type error) and must treat results as **immutable** (copy before edit).

```tsx
const setStatus = useMutation(api.housekeeping.setStatus).withOptimisticUpdate(
  (store, { taskId, status }) => {
    const cur = store.getQuery(api.housekeeping.listByAssignee, { status: "pending" });
    if (cur) store.setQuery(api.housekeeping.listByAssignee, { status: "pending" },
      cur.map(t => t._id === taskId ? { ...t, status } : t));
  });
```

Use for instant housekeeping status toggles / order-lane moves (perceived latency on poor mobile networks, NFR low-bandwidth driver).

### PWA / offline (Serwist + Story 1.6 indicator + deferred mutation queue)

- **Indicator (Story 1.6):** keep it. Drive the online/offline banner from `useConvexConnectionState().isWebSocketConnected` (plus `navigator.onLine`) rather than a custom socket ping.
- **What Convex does for offline mutations:** a `useMutation` call while disconnected returns a pending Promise; the request is held **in-memory** in the client's `RequestManager` and **resent on reconnect** (`restart()`/`resume()` in the verified sources). So in-session optimistic UI + auto-replay works for free.
- **What Convex does NOT do — still needs custom handling:** the in-flight queue is **in-memory only**; there is **no IndexedDB/localStorage persistence**. A full page reload or app kill while offline (the realistic caretaker/housekeeping scenario, NFR2/NFR3/NFR5) **loses unsent mutations**. The Serwist Background Sync queue does **not** help here — Convex mutations go over WebSocket, not interceptable `fetch`. The deferred "offline mutation queue" must therefore be a **custom durable queue** (e.g. IndexedDB) for staff write-while-offline flows; replay on reconnect by calling the same mutation. Convex transactions are single-document-consistent and last-write-wins, so the custom layer also owns **conflict resolution** (architecture.md lists "offline sync & conflict resolution" as a core concern). Per the Addendum, revisit this "when the first mutations land."

### Gotchas / what to verify

- `useQuery` returning `undefined` means *loading*, not empty — never conflate with empty arrays.
- Optimistic update handlers must be **sync + immutable**; mutating cached results corrupts the client store.
- Serwist must **not** try to cache/intercept the Convex WebSocket (`.convex.cloud`); offline writes are out of scope for the SW.
- `_generated/api` + `_generated/dataModel` must exist (run `npx convex dev`/`codegen`) before imports resolve; verify the import path from `apps/web` to `packages/backend/convex/_generated`.
- Verify the WebSocket survives the Next.js 16 + React 19 setup with a single long-lived `ConvexReactClient` instance (don't recreate per render).

## Scheduler & Crons

**Convex version:** `convex@1.40.0`. API confirmed from the package's `dist/esm-types/server/scheduler.d.ts` and `cron.d.ts`. This replaces Redis + BullMQ per the architecture Backend Platform Addendum (the "Async / jobs" row and AR3′).

### Recommended patterns

**One-shot / delayed jobs** — `ctx.scheduler` is available in every `mutation` and `action`:

```ts
// in a mutation/action handler
await ctx.scheduler.runAfter(0, internal.payments.reconcile, { paymentId });      // ASAP after commit
await ctx.scheduler.runAfter(120_000, internal.payments.expireStkPush, { paymentId }); // +2 min
await ctx.scheduler.runAt(Date.now() + 30_000, internal.notifications.send, { id }); // absolute time
// returns Id<"_scheduled_functions"> — store it to cancel later:
await ctx.scheduler.cancel(scheduledId);
```

**Recurring jobs** — a single `packages/backend/convex/crons.ts` (file currently absent — must be created), default-exporting `cronJobs()`:

```ts
import { cronJobs } from "convex/server";
import { internal } from "./_generated/api";

const crons = cronJobs();
crons.daily("nightly export", { hourUTC: 21, minuteUTC: 0 }, internal.reports.dailyExport, {}); // 00:00 EAT
crons.interval("low-stock scan", { minutes: 30 }, internal.inventory.lowStockScan, {});
crons.cron("backup trigger", "0 21 * * *", internal.backups.run, {}); // raw 5-field cron
export default crons;
```

Helpers: `interval` ({seconds|minutes|hours}), `hourly`, `daily`, `weekly`, `monthly`, `cron` (raw string). **All times are UTC** — Kenya is UTC+3, so convert (e.g. midnight EAT = `hourUTC: 21` the prior day). Targets must be `mutation`/`action`, public or internal; prefer `internalMutation`/`internalAction` so clients can't invoke them directly.

### SommyComfort mapping (data-model.md / AR addenda)

- **M-Pesa callback timeout** (AR7′; `Payment` entity, `status: pending → confirmed/failed`): when the STK push action fires, schedule `runAfter(~120s, internal.payments.expireStkPush, { paymentId })`. The HTTP-action callback (`convex/http.ts`) confirms the payment and should `cancel()` the timeout; if the callback never arrives, the timeout mutation flips `status` to `failed`. Replaces the BullMQ "payment callback" queue.
- **Notification fan-out** (`NotificationLog` channel = email|sms|whatsapp|push): a mutation enqueues one `runAfter(0, internal.notifications.deliverOne, { logId })` per recipient/channel; each delivery is an `action` (external API call) that updates `status: queued → sent|failed`.
- **Report generation / daily backup/export**: `crons.daily(...)` → an `internalAction` that aggregates and writes a Convex file (`ctx.storage`) or pushes an export. Replaces BullMQ PDF/CSV jobs.
- **Low-stock alerts** (R2/R3, deferred; `InventoryItem.reorderLevel`, `StockMovement`): `crons.interval` scans items below `reorderLevel` and schedules notifications.

### Durability / idempotency

- **Execution guarantees (from the installed types):** scheduled **mutations run exactly-once** (auto-retried on transient errors); scheduled **actions run at-most-once** (NOT retried). So put critical/idempotent state transitions in mutations; use actions only for the external side-effect, and have the action schedule a mutation to record the result.
- Because actions can silently fail and cron'd/at-least-once flows can double-fire, make all handlers **idempotent**: guard on current state (e.g. only act if `Payment.status === "pending"`), and lean on the existing `@@unique([provider, providerReceiptNumber])` dedupe on `Payment` (re-express as a Convex index + uniqueness check inside the mutation — Convex has no DB-level unique constraint).
- Jobs land in the `_scheduled_functions` system table; cancellation of a not-yet-started job prevents the run, and is transactional for mutations.

### Gotchas / what to verify

- `crons.ts` does not yet exist under `packages/backend/convex/` — create it; only **one** default-exported `cronJobs()` per deployment.
- `runAfter`/`runAt` are only on `ctx` in mutations/actions — **not** in queries.
- `runAt` timestamp must be within ±5 years; `runAfter` delay must be ≥ 0.
- `crons.monthly` with `day > 28` skips short months (e.g. won't run in February) — relevant for any monthly financial report.
- Verify cron + scheduler behavior against the dev deployment `quixotic-boar-465` and unit-test with `convex-test`. Cron registration/changes take effect on `convex deploy`.

**Recommended pattern** — `convex-test` provides an in-memory mock of the Convex backend (`convexTest`) that runs against your real `schema` + functions, driven by Vitest under the `edge-runtime` environment. Functions are loaded via `import.meta.glob`.

```ts
// packages/backend/convex/auditLogs.test.ts
import { convexTest } from "convex-test";
import { expect, test } from "vitest";
import schema from "./schema";
import { api } from "./_generated/api";
// Vitest needs the glob so convex-test can find all function modules:
const modules = import.meta.glob("./**/!(*.*.*)*.*s");

test("record + listForEntity round-trips", async () => {
  const t = convexTest(schema, modules);
  await t.mutation(api.auditLogs.record, {
    action: "booking.create", entityType: "booking", entityId: "b1",
  });
  const rows = await t.query(api.auditLogs.listForEntity, {
    entityType: "booking", entityId: "b1",
  });
  expect(rows).toHaveLength(1);
});

test("health.check is ok", async () => {
  const t = convexTest(schema, import.meta.glob("./**/!(*.*.*)*.*s"));
  expect(await t.query(api.health.check, {})).toEqual({ status: "ok" });
});
```

- **Queries/mutations:** `t.query(...)` / `t.mutation(...)`. **Actions:** `t.action(...)`; external `fetch`/scheduler calls inside actions should be mocked with `vi.stubGlobal`/`vi.mock` (relevant later for M-Pesa actions, AR7′).
- **Faking auth/identity:** use `t.withIdentity({ subject, name, ... })` to get a scoped client whose `ctx.auth.getUserIdentity()` returns that identity — e.g. `await t.withIdentity({ subject: "user|123" }).mutation(api.auditLogs.record, {...})`. This is how you'll exercise the `requirePermission(ctx, area, action)` RBAC helper (AR6′) and the per-actor audit write once `actorId` derives from `ctx.auth` instead of args.

**Setup steps for `packages/backend` (do after `convex/_generated` exists — i.e. after first `convex dev`/`codegen`):**

2. Add a `test` script to `packages/backend/package.json`: `"test": "vitest run"` (and `"test:watch": "vitest"`). This makes the root `turbo run test` pick it up automatically.
3. Create `packages/backend/vitest.config.ts`:

   ```ts
     test: {
       environment: "edge-runtime",

     },
   });
   ```
