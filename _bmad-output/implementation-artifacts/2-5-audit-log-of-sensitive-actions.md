# Story 2.5: Audit log of sensitive actions

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an admin/owner,
I want a tamper-evident log of sensitive actions,
so that I can investigate and meet accountability needs.

> **Scope (audit substrate + admin viewer).** This story hardens the **already-scaffolded** `auditLogs` table + functions (`packages/backend/convex/auditLogs.ts`, `schema.ts`) into the cross-cutting audit substrate (AR9) the whole platform depends on: it (a) makes the audit row **actor-authoritative** (actor derived from `ctx.auth`, not client args) now that Convex Auth (Story 2.1) + RBAC (Story 2.3) exist, (b) adds a **filterable, paginated admin viewer query** gated by `requirePermission`, (c) provides a **shared `writeAudit` helper** so every state-changing mutation in Epics 2–9 writes a consistent, atomic audit row, and (d) wires it into the **already-existing sensitive mutations** in the Epic 2 surface (auth, staff, roles/permissions). It is **append-only by design** ("tamper-evident" = no update/delete function is exposed; immutability via Convex's lack of an update/delete path here, not cryptographic hash-chaining — see Dev Notes for the honest interpretation).
>
> **Right-scope caveat (binding).** `@convex-dev/auth` and `convex-test` are **not yet installed**, `convex/_generated/*` is **not generated**, and a live `convex dev`/`codegen`/deploy needs a Convex login that is **unavailable in this dev environment**. So the function/schema/helper/test source and the web viewer are **buildable-offline as source**; codegen-dependent typecheck, the live reactive query, and seeded-data verification are **deferred to first `convex dev`** (see Dev Notes "Buildable-offline vs deferred"). This mirrors Stories 2.1–2.4.

## Acceptance Criteria

1. **Sensitive mutations write an audit row (actor-authoritative).** (FR17, NFR8, AR9; reframed from the epic's `audit_log`-row AC) — Every money-affecting or sensitive mutation in the existing Epic 2 surface (auth sign-in/sign-out events available to the app, staff create/edit/deactivate, role create/rename, permission grant/revoke) writes **exactly one** `auditLogs` row **in the same transaction** as the change. The row records `actorId` (derived from `ctx.auth` via `getAuthUserId`, **never** from client args), `action` (a `"<area>.<verb>"` string, e.g. `staff.deactivate`, `role.create`, `permission.revoke`), `entityType`, `entityId`, optional `before`/`after` snapshots, optional `ip`, and the timestamp (Convex's built-in `_creationTime`). A shared `writeAudit(ctx, {...})` helper in `convex/lib/audit.ts` is the single write path; ad-hoc `ctx.db.insert("auditLogs", …)` is removed in favor of it.
2. **Append-only / tamper-evident.** No public or internal Convex function exposes an `update` or `delete` over `auditLogs`; the only write path is the append-only `record`/`writeAudit`. A test asserts no audit-mutation API surface allows modification (i.e. the audit module exports only the append + read functions). Retention/deletion, if ever needed, is an explicitly out-of-scope future admin action.
3. **Admin viewer query — filter by actor, entity, date range.** (epic AC2) — A `listAudit` query in `auditLogs.ts`, gated by `requirePermission(ctx, "Audit", "read")`, returns audit rows **newest-first**, supporting optional filters: `actorId`, `entityType` (+ optional `entityId`), and a `dateRange` (`{ from?: number; to?: number }` over `_creationTime`). It is **paginated** (Convex `paginationOpts` / `.paginate(...)`), so large logs don't over-read. Unfiltered and each single-filter combination return correct, time-ordered results. Unauthorized callers (`requirePermission` fails) are rejected with `FORBIDDEN: Audit:read`.
4. **Export of matching entries.** (epic AC2 "exportable") — The same filter inputs feed an export path: a `listAuditForExport` query (or a bounded variant of `listAudit`) returns the filtered rows as a flat, serializable array suitable for CSV, and the web viewer offers a **"Export CSV"** action that downloads the currently-filtered result client-side (columns: timestamp, actor, action, entityType, entityId, ip). Export respects the same `requirePermission(ctx, "Audit", "read")` gate (server-authoritative; the client never sees rows it isn't permitted to read).
5. **Admin web viewer surface.** A read-only audit page lives under the Admin workspace (`apps/web/src/app/(app)/admin/audit/…`), using `useQuery(api.auditLogs.listAudit, …)` (reactive — new sensitive actions appear without reload, NFR10), with filter controls (actor / entity / date range), a paginated table, the Export CSV button, and the established loading (`undefined`)/empty/error states. Nav visibility is gated by the caller's `Audit:read` permission via the existing `myPermissions` UI gate (Story 2.3) — server enforcement in AC3 remains authoritative.
6. **Schema correctness for the actor relation + query indexes.** `auditLogs.actorId` is migrated from `v.optional(v.string())` to `v.optional(v.id("users"))` (now that the `users` table exists from Story 2.1), and the table carries indexes supporting the viewer filters: the existing `by_entity` (`entityType`,`entityId`) and `by_actor` (`actorId`), kept/extended so actor and entity filters use an index rather than a full scan; date-range-only queries fall back to `_creationTime` ordering. Existing `auditLogs.record`/`listForEntity` continue to work (no regression).
7. **Tests + green.** `convex-test` tests (colocated `auditLogs.test.ts`) cover: a `writeAudit` round-trip derives `actorId` from `withIdentity` (not args); `listAudit` returns newest-first and filters correctly by actor, by entity, and by date range; pagination returns a cursor and the page size is respected; an unauthorized identity is rejected by the `requirePermission` gate; and no update/delete path exists (AC2). Web: an RTL test renders the viewer with a mocked `useQuery` (filters render, rows list, Export button present). Where these depend on `_generated`/installed packages, they are written now and **run after first `convex dev`** (documented, consistent with 2.1–2.4). The web gates (`pnpm typecheck/lint/test/build`) stay green for the buildable-offline web surface.

> Out of scope: cryptographic hash-chaining / WORM storage (the "tamper-evident" interpretation here is append-only-by-API, not cryptographic — note this honestly); retention/purge jobs; auditing mutations from Epics 3–9 (those stories call the shared `writeAudit` helper as they land — this story provides the helper + wires only the **existing** Epic 2 mutations); server-side PDF export (CSV only here); diff/visualization of `before`/`after` beyond raw display.

## Dev Notes

### Build on the existing scaffold (do not rebuild)
- **Source-of-truth files already present:** `packages/backend/convex/auditLogs.ts` (`record` mutation + `listForEntity` query) and `packages/backend/convex/schema.ts` (`auditLogs` table with `by_entity` + `by_actor` indexes; `actorId` currently `v.optional(v.string())` with a banner comment "→ `v.id("users")` once the users table lands"). This story executes that banner. [Source: packages/backend/convex/auditLogs.ts, packages/backend/convex/schema.ts]
- **Convention (Convex Implementation Guide):** "Audit is written from mutations — every state-changing mutation writes an `auditLogs` row within the same transaction, so the audit trail is atomic with the change. `auditLogs.actorId` derives from `ctx.auth`, not from client args." This story makes that real and provides the shared helper so Epics 3–9 follow it uniformly.

### Convex specifics — tables / functions / validators / RBAC
- **Table:** `auditLogs` (the single AR9 cross-cutting table). Fields: `actorId: v.optional(v.id("users"))`, `action: v.string()`, `entityType: v.string()`, `entityId: v.optional(v.string())`, `before: v.optional(v.any())`, `after: v.optional(v.any())`, `ip: v.optional(v.string())`. Timestamp is the built-in `_creationTime` (no manual `createdAt`). Indexes: `by_entity` (`entityType`,`entityId`), `by_actor` (`actorId`). Maps to data-model.md `AuditLog` (`actor_id`, `action`, `entity_type`, `entity_id`, `before Json?`, `after Json?`, `ip`, `@@index([entityType,entityId])`, `@@index([actorId,createdAt])`) — the Prisma `@@index([actorId, createdAt])` becomes `by_actor` + `_creationTime` ordering. [Source: _bmad-output/planning-artifacts/data-model.md lines 111–124]
- **Functions:** keep `record` (now actor-from-auth) + `listForEntity`; add `listAudit` (paginated, filtered, `requirePermission(ctx,"Audit","read")`) and `listAuditForExport` (bounded). The shared write path is `lib/audit.ts#writeAudit`. Prefer the public `query` for the viewer (reactive `useQuery`), the helper is plain TS (not a registered function).
- **RBAC (AR6′):** the viewer/export are gated by `await requirePermission(ctx, "Audit", "read")` as the first line (helper from `convex/lib/auth.ts`, Story 2.3). "Audit" must be one of the 18 permission areas (PRD §7) — confirm the exact area label used by the Story 2.3 seed catalog and reuse that literal (do not invent a new area). `manage`/`write` over audit is not used here (append-only); only `read` gates the viewer. [Source: Convex Implementation Guide §Auth&RBAC; _bmad-output/implementation-artifacts/2-3-roles-and-granular-permissions-rbac.md]
- **Validators are the contract:** declare `args` with `convex/values` for every function; `actorId`/`entityId` filters are `v.optional(v.id("users"))` / `v.optional(v.string())`; date bounds `v.optional(v.number())` (epoch ms over `_creationTime`); pagination via Convex `paginationOpts`. The web mirrors these for UX only and is never trusted.
- **Reactivity (NFR10):** `listAudit` is a Convex query → `useQuery` is a live subscription; a new sensitive action elsewhere appears in the open audit viewer within the ≤5s target without polling. [Source: Convex Implementation Guide §Realtime]
- **Tamper-evident — honest interpretation:** Convex has no DB-level immutability/WORM and this story does **not** add cryptographic hash-chaining. "Tamper-evident" here means **append-only by API**: no `update`/`delete` function over `auditLogs` is exposed, so there is no application path to alter history; `actorId` is server-derived so an attacker can't forge the actor. A future hardening (hash chain / external WORM sink) is out of scope and should be tracked. State this in Completion Notes.
